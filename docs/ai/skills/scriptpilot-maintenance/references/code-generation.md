# 代码生成规范

## 通用规则

- 修改前先读相关模块，保持现有分层和命名风格。
- 手工编辑使用小范围补丁，不做无关重构。
- 默认使用 ESM、原生 Node API 和项目已有工具，不额外引入依赖，除非有明确收益。
- 不把 CK、日志、缓存、临时数据、用户数据写入提交或最终说明。
- 新功能优先做到端到端真实可用，再补 UI 反馈和错误处理。

## Renderer

- Renderer 是原生 HTML/CSS/JS，不使用 React。
- 高频列表操作优先使用事件委托，避免每次渲染为大量按钮重新绑定监听器。
- 搜索、过滤、输入联动要节流或合并渲染，避免长列表输入卡顿。
- 表格排序必须有可见图标或状态，让用户知道当前可排序和排序方向。
- 日志、脚本选择、订阅日志等大内容优先使用弹窗展示，不在列表页堆叠长文本。
- 选择已有脚本创建任务时，不展示脚本内容输入框；新建脚本文本时才展示内容输入。

## Main 与模块

- 业务能力放在 `src/main/modules/*`，Electron 桥接只做调用和编排。
- 命令型操作走 command handler，查询型操作走 query handler，避免 UI 直接操作 repository。
- 文件写入必须经过便携目录解析，不写 AppData 或用户目录。
- 长耗时操作要尽早返回可轮询状态，避免阻塞 UI。
- 可并行的 IO 扫描使用目录级并发，但要避免无上限并发压垮磁盘。

## 青龙式能力

- 订阅、脚本、环境变量、配置、依赖的交互优先参考青龙面板的用户心智。
- 订阅自动建任务只基于脚本内明确可解析的 cron 声明，没有 cron 或 cron 无效时跳过，不猜测。
- 脚本选择使用树型弹窗，和脚本管理页保持一致。
- 任务日志和订阅日志都用弹窗查看，列表只显示状态、时间和操作入口。

## 验证

- JS 文件改动后运行 `node --check <file>`。
- UI 改动需要验证源码文件和 release resource 副本。
- 数据路径相关改动必须用隔离 `portableRoot` 或隔离 Electron `cwd` 验证。
- 发布前确认 `release/win-unpacked/app/data/` 不被覆盖、不被打包。
